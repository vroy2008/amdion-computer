# Amdion V1 — The Simplified Spine

*The single source of truth for what V1 is, what it is not, and how the code maps to it. Written 2026-06-21. Supersedes the broader surface described in [REORIENTATION.md](REORIENTATION.md) — that doc remains the rationale; this doc is the scope.*

---

## 1. The overarching goal

**Amdion helps you see what you're doing on your computer and manage your attention — so you're driving, not being driven. That's it.**

Everything in V1 serves that one sentence. Anything that doesn't is *bonus* — kept in the repo, gated dormant, rolled out later.

The product is two halves that connect:

- **Desktop app — Sense.** Mark sessions, capture intent, keep an honest log, let you see it. Passive now; the intent labels become active levers later.
- **Chrome extension (Defend) — Act.** Track, Nudge, or Block — driven by the desktop's intent when connected, standalone-capable when not.

```
Desktop app  ──intent / friction──▶  Extension (Defend)
   Sense          localhost WS           Act
 (log/review)   ◀──activity/track──   (off/nudge/block)
```

## 2. Desktop app

### 2.1 Session sensing
- A session boundary is derived from **screen state** (wake / lock / unlock + idle) — **never** computer power on/off.
- Time-aware: elapsed time from a **monotonic clock**; wall-clock used only for "new day / morning." A clock change or sleep-across-midnight must not corrupt a session.
- One session boundary, shared by the greeting, the log, and the classifier (`classify()` already owns this).

### 2.2 Intent
- Three defaults: **Deep work · Communication · Exploration**. User can add/edit.
- Chosen at session start *or* picked later. **Late capture:** if a session started with no intent and the user picks one mid-session, record it against the running session.
- V1 = passive logging. Future = drives the experience (already true for the extension; see §3.2).

### 2.3 Log
- Append session + intent + merged activity (OS sensing ∪ browser track) to the local SQLite store.
- **Exportable** from the app (CSV / JSON) — small net-new.

### 2.4 Review
- One calm surface ("let them see this") — the existing Observer view, kept and made the home of review.

### 2.5 Onboarding — 4 slides (keep as-is)
0. Welcome (logo hero + summon hint)
1. Optional Mac tuning (3 reversible tweaks)
2. Connect Chrome (load unpacked from the bundle)
3. Done (it's in your menu bar)

### 2.6 Panel — only these
- Greeting + **intent** (pick at start / active / late-capture).
- **Session running** status.
- A **collapsed "recommended Mac tweaks" disclosure** behind a hint (the 3 tweaks, surfaced from the panel — not buried in a full Settings overlay).
- **No mode toggle in the panel** — mode is expressed via intent (see §3.2). Everything else in today's Settings overlay is parked.

## 3. Extension (Defend)

### 3.1 Three modes (the spine)
Map 1:1 onto the existing friction levels — a rename/reframe, not new machinery:

| Mode | Mechanism | Behavior |
|------|-----------|----------|
| **Off** | `off` | track only |
| **Nudge** | `soft` | in-page nudge card on the distraction set |
| **Block** | `lockin` | DNR redirect to `blocked.html` |

Tracking runs in **all three** modes — track is the always-on baseline. The user-facing toggle lives in the **extension action** (simple terms: Off / Nudge / Block).

The **distraction set** = built-ins (`youtube, x/twitter, facebook, instagram, reddit, tiktok, netflix, twitch`) ∪ the user's block list pushed from the app.

### 3.2 The intent ↔ mode contract (load-bearing)
- **Intent is the default writer.** When the app is connected and intent is set, it pushes the mapped mode:
  - Deep work → **Block**
  - Communication → **Nudge**
  - Exploration → **Off**
- **The manual toggle is a session-scoped override.** A new session, or the next intent change, re-asserts the intent default. (This also means an app reconnect re-pushing the intent mode is *expected*, not a clobber.)
- **Disconnected / no app:** default mode = **Nudge** (Defend is gently-on out of the box). Last manual choice sticks until changed. Block stays opt-in (intent = Deep work, or manual) — never hard-block before the app is even connected.
- **No intent yet, app connected:** Off (track only) until intent is chosen.

### 3.3 Distribution — app-first
- **V1:** open-source repo, **build-from-source** onboarding. Clone → build app → load extension unpacked (today's Step 2). Locally-built app isn't quarantined → no Gatekeeper prompt. Audience = developers / early adopters.
- An optional unsigned GitHub Release is possible, but on **macOS 15 (Sequoia)** an unsigned download requires *System Settings → Privacy & Security → "Open Anyway."* Lead with build-from-source.
- **Demand-phase upgrade:** Apple Developer ($99/yr) → signed/notarized DMG + auto-updater; Chrome Web Store ($5 once) → one-click install. Extension ID is already pinned via `key` in the manifest. Expect Web-Store review to question `<all_urls>` + `declarativeNetRequest` + the localhost bridge.

## 4. The bonus shelf — kept, gated, maintained independently

Read Mode, Notes/Capture, Present, and the whole Reshape layer (declutter / feed-fade / YT-drift) are **not V1 core**. They are **not deleted** — they stay in the repo, gated dormant, each unlocked explicitly with a brief per-feature onboarding before first use.

**Isolation model** (so a feature can be edited without risking the core):
- **Feature-module layout** — `extension/core/` (bridge, track, nudge, block) + `extension/features/{read-mode,notes,present,reshape}/`, each self-contained.
- **Unlock gate** — each feature reads an `unlocked` flag; until set, it no-ops (fails *closed* — a broken feature can't take down core).
- **Namespaced globals** — same-extension content scripts share one isolated world per frame, so each feature namespaces under `window.__amdion.<feature>` and guards with try/catch.
- **Registration** — prefer runtime injection (`chrome.scripting.registerContentScripts`) on unlock for true dormancy; the interim acceptable version is "in manifest but self-no-op unless unlocked."
- **MV3 constraint to accept:** one manifest, one service worker — *editing* an existing feature is fully isolated; *adding* a new one still touches one registry line. That's the realistic version of "independent."

## 6. Next steps — one branch, ordered commits

All V1 work lands on a **single line** (continue `feat/attention-capture-notes`, or merge it to master and work there) as incremental commits, in this order. **Feature isolation here means the code layout (`core/` + `features/`), not git branches** — it holds within one branch.

1. **Feature isolation** — establish `core/` + `features/` layout, the unlock-gate registry, namespaced globals. Move bonus features into modules (no behavior change yet). *Do first — unblocks the rest.*
2. **Extension modes** — Off/Nudge/Block action toggle, intent→mode mapping + override precedence (§3.2), default **Nudge** when disconnected. Trim `background.js`/`manifest.json` to core.
3. **Panel simplify** — panel = greeting + intent (+late-capture) + session + collapsed Mac-tweaks; remove mode toggle and Settings sprawl.
4. **Log export** — CSV/JSON export from the Observer/review surface.
5. **Packaging & README** — app structure + how it unpacks for new users (build-from-source, extension folder in the bundle), rewrite README for app-first open-source V1. *Last — describes the final shape.*
6. **Verify** — real-Chrome smoke test of Off/Nudge/Block + intent-driven switching (env can't run MV3).

Each step is one (or a few) self-contained commits, app building green between them. Bonus rollout (later, per feature): unlock flag + mini-onboarding + runtime registration.
